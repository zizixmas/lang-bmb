# BMB Security Audit Checklist

> v1.0.0-beta 릴리스 전 필수 보안 검토 체크리스트

---

## Overview

이 문서는 BMB 컴파일러와 런타임의 보안을 검토하기 위한 체크리스트입니다.
모든 항목은 v1.0.0-beta 릴리스 전에 검토되어야 합니다.

### 심각도 분류

| 레벨 | 설명 | 릴리스 블로커 |
|------|------|--------------|
| 🔴 Critical | 원격 코드 실행, 권한 상승 | Yes |
| 🟠 High | 정보 유출, DoS 가능 | Yes |
| 🟡 Medium | 제한적 영향 | Depends |
| 🟢 Low | 미미한 영향 | No |

## 1. 컴파일러 입력 검증

### 1.1 소스 코드 파싱

| 항목 | 심각도 | 상태 | 검증 방법 |
|------|--------|------|----------|
| 무한 재귀 문법 방지 | 🟠 High | [x] | 깊이 제한 있음 ("stack overflow" 오류) |
| 대용량 파일 처리 | 🟡 Medium | [ ] | 100MB+ 파일 테스트 |
| 잘못된 UTF-8 처리 | 🟡 Medium | [x] | null bytes 거부됨 ("unexpected character") |
| 주석 내 특수 문자 | 🟢 Low | [ ] | 특수 문자 테스트 |

**테스트 명령:**
```bash
# 재귀 깊이 테스트
echo "fn f() = f();" | timeout 5 bmb check -

# 대용량 파일
dd if=/dev/zero bs=100M count=1 | bmb check -

# Fuzzing
cargo fuzz run parser
```

### 1.2 토큰화

| 항목 | 심각도 | 상태 | 검증 방법 |
|------|--------|------|----------|
| 정수 오버플로우 리터럴 | 🟠 High | [x] | i64::MAX+1 테스트 (wrap 동작 확인) |
| 문자열 길이 제한 | 🟡 Medium | [ ] | 10MB 문자열 테스트 |
| 특수 이스케이프 시퀀스 | 🟡 Medium | [ ] | 모든 이스케이프 테스트 |

**테스트 케이스:**
```bmb
// 오버플로우 테스트
let x = 9223372036854775808;  // i64::MAX + 1

// 긴 문자열
let s = "..." // 10MB 문자열
```

## 감사 수행 절차

### Phase 1: 자동화 검사 (1-2일)

```bash
# 1. 정적 분석
cargo clippy --all-targets -- -D warnings

# 2. Fuzzing (24시간)
cargo fuzz run parser -- -max_total_time=86400

# 3. 메모리 검사
cargo +nightly miri test

# 4. 의존성 취약점
cargo audit
```

### Phase 2: 수동 검토 (2-3일)

1. **컴파일러 코드 리뷰**
   - `bmb/src/codegen/llvm.rs` - LLVM IR 생성
   - `bmb/src/interp/` - 인터프리터
   - `bmb/src/smt/` - SMT 검증

2. **런타임 코드 리뷰**
   - `runtime/` - C 런타임 함수
   - 메모리 관리 코드

3. **Bootstrap 컴파일러 리뷰**
   - `bootstrap/llvm_ir.bmb` - LLVM IR 생성

### Phase 3: 침투 테스트 (1-2일)

1. 악의적 BMB 소스 코드 작성
2. 컴파일러 크래시 시도
3. 메모리 손상 시도
4. 정보 유출 시도

## 검증 결과 (2026-01-14)

### 1차 자동화 검사 결과

| 검사 | 결과 | 상세 |
|------|------|------|
| `cargo clippy --all-targets` | ✅ 통과 | 0 경고 |
| `cargo audit` | ⏳ 미실행 | cargo-audit 미설치 |
| `@trust` 사용 현황 | ✅ 0개 | bootstrap/, stdlib/, examples/ 검색 |
| `unsafe` Rust 코드 | ⚠️ 28개 블록 | bmb/src/interp/eval.rs (메모리 관리) |

### 정수 오버플로우 테스트

| 테스트 | 결과 | 설명 |
|--------|------|------|
| i64::MAX + 1 | wrap → i64::MIN | 의도된 동작 (문서화됨) |
| 1e12 * 1e12 | wrap | 의도된 동작 (문서화됨) |

**결론**: 정수 오버플로우는 wrap 방식으로 처리됨. `@check` 어노테이션으로 검증 가능.

### 재귀 깊이 테스트

| 테스트 | 결과 | 설명 |
|--------|------|------|
| 10,000 호출 | ✅ 정상 | 빠르게 완료 |
| 100,000 호출 | ✅ 오류 처리 | "stack overflow: too deep recursion" 메시지 |

**결론**: 스택 오버플로우가 안전하게 처리되고 명확한 오류 메시지 출력.

### unsafe Rust 코드 검토 필요

`bmb/src/interp/eval.rs`의 unsafe 블록 목록:
- Line 1796, 1840, 1865, 1930: `std::alloc::alloc` (메모리 할당)
- Line 1887, 1908, 1935: 포인터 역참조
- Line 1965, 1986, 1995, 1998, 2007: Vec/HashMap 내부 메모리
- Line 2029-2592: 런타임 함수 메모리 연산

**상태**: ✅ Phase 2 완료 (아래 참고)

### 2차 수동 검토 결과 (Phase 2)

#### 메모리 할당 함수 (malloc, free, realloc, calloc)

| 함수 | 결과 | 분석 |
|------|------|------|
| `builtin_malloc` | ✅ 안전 | null 체크, Layout 검증, zeroed 할당 |
| `builtin_free` | ✅ 안전 | 의도적 메모리 누수 (인터프리터 안전성) |
| `builtin_realloc` | ✅ 안전 | null 체크, 새 메모리 할당 후 복사 |
| `builtin_calloc` | ✅ 안전 | 오버플로우 체크 (`checked_mul`) |

**참고**: `free()`가 실제로 메모리를 해제하지 않는 것은 인터프리터 모드의 의도된 동작.
네이티브 컴파일 시에는 LLVM 백엔드가 실제 dealloc 수행.

#### Vec 연산

| 함수 | 결과 | 분석 |
|------|------|------|
| `vec_get` | ✅ 안전 | 경계 검사 (`idx < 0 || idx >= len`) |
| `vec_set` | ✅ 안전 | 경계 검사 + 에러 메시지 |
| `vec_push` | ✅ 안전 | 용량 초과 시 2배 확장, null 체크 |
| `vec_pop` | ✅ 안전 | 빈 Vec 검사 |

**결론**: 모든 Vec 연산에 적절한 경계 검사 존재.

#### HashMap 연산

| 항목 | 결과 | 분석 |
|------|------|------|
| null 체크 | ✅ | 모든 함수에서 null map 검사 |
| 로드 팩터 | ✅ | 70% 제한으로 충돌 DoS 완화 |
| 메모리 해제 | ✅ | `hashmap_free`에서 모든 배열 정리 |
| 해시 함수 | ⚠️ | FNV-1a 기반 (단순하지만 적절) |

**참고**: 해시 함수가 단순하지만, 로드 팩터 제한으로 해시 충돌 DoS 위험 완화됨.

#### 파일 I/O 연산

| 함수 | 결과 | 분석 |
|------|------|------|
| `read_file` | ⚠️ 문서화됨 | 경로 순회 허용 (시스템 언어 특성) |
| `write_file` | ⚠️ 문서화됨 | 경로 순회 허용 (시스템 언어 특성) |
| `file_exists` | ✅ | 읽기 전용, 낮은 위험 |

**결론**: 파일 I/O는 OS 권한에 의존. 샌드박싱은 WASM 또는 컨테이너에서 수행.

#### String 연산

| 항목 | 결과 | 분석 |
|------|------|------|
| 할당 | ✅ | Rust `Rc<String>` 사용 (안전) |
| UTF-8 | ✅ | Rust 표준 라이브러리 사용 |
| 해제 | ✅ | Rust RAII로 자동 관리 |

**결론**: 문자열은 Rust의 안전한 메모리 관리 사용.

| 많은 let 바인딩 (100개) | ✅ 안전 | 정상 처리 |
| 0으로 나눔 | ✅ 안전 | "division by zero" 오류 |
| 0으로 모듈로 | ✅ 안전 | "division by zero" 오류 |

#### 컴파일러 크래시 시도 결과

| 테스트 | 결과 | 설명 |
|--------|------|------|
| 잘못된 이스케이프 | ✅ 안전 | 렉서 오류 |
| 닫히지 않은 문자열 | ✅ 안전 | 렉서 오류 |
| 닫히지 않은 주석 | ✅ 안전 | 파서 오류 (/* 미지원) |
| 바이너리 데이터 (null bytes) | ✅ 안전 | "unexpected character: \"\\0\"" 오류 |
| 빈 파일 | ✅ 안전 | 정상 처리 |
| 공백만 있는 파일 | ✅ 안전 | 정상 처리 |
| 순환 타입 에일리어스 (미사용) | ✅ 통과 | 타입 체크 통과 (지연 평가) |
| **순환 타입 에일리어스 (사용)** | 🟠 DoS | **무한 루프 (timeout 발생) - 수정 필요** |
| 깊은 튜플 타입 | ✅ 안전 | 파서 오류 |
| 빌트인 함수 섀도잉 | ⚠️ 허용됨 | 사용자 정의 함수가 우선 (검토 필요) |
| 중복 main 함수 | ⚠️ 허용됨 | 마지막 정의 우선 (경고 필요) |

#### 발견된 문제점

##### 🟠 High: 순환 타입 에일리어스 DoS (CVE 후보)

```bmb
type A = B;
type B = A;
fn test(x: A) -> A = x;  // 이 줄에서 무한 루프 발생
```

- **심각도**: 🟠 High (DoS 가능)
- **영향**: 악의적 소스 코드로 컴파일러 무한 루프 유발 가능
- **권장 수정**: 타입 해석 시 순환 감지 로직 추가, 최대 깊이 제한

##### 🟡 Medium: 중복 함수 정의 허용

```bmb
fn main() -> i64 = 1;
fn main() -> i64 = 2;  // 두 번째 정의가 첫 번째를 덮어씀
```

- **심각도**: 🟡 Medium (혼란 유발)
- **영향**: 코드 의도와 다른 동작 가능
- **권장 수정**: 중복 함수 정의 시 컴파일 오류 또는 경고 발생

##### 🟢 Low: 빌트인 함수 섀도잉

```bmb
fn malloc(n: i64) -> i64 = 0;  // 빌트인 malloc 섀도잉
```

- **심각도**: 🟢 Low (의도적일 수 있음)
- **영향**: 시스템 언어에서 빌트인 대체는 유효한 패턴
- **권장 조치**: 경고 메시지 추가 (선택적)

#### Phase 3 결론

- **크리티컬 문제 없음**: 컴파일러 크래시, 메모리 손상, 정보 유출 시도 모두 방어됨
- **수정 필요**: 순환 타입 에일리어스 DoS 문제
- **개선 권장**: 중복 함수 정의 경고/오류, 빌트인 섀도잉 경고
